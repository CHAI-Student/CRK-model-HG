# 0713 — held-object(지속 노출) 강등 설계 (이슈 #10 A안)

> 2026-07-13. 이슈 #10 분석의 대응안 A를 구체화한 설계.
> 상태: **설계만** — 구현 전. 역효과 시나리오(§7) 검토 포함.

## 1. 배경 — 0711 창 모델이 못 잡는 새 실패 모드

이슈 #10: zone2에서 취출한 체리쥬빌레 2개(class 40, 131g)를 반대 손에
**계속 들고** zone1에서 메로나+쿠즈락을 취출 → zone1 영상 내내 체리쥬빌레가
잡혀 225표(ratio 0.54)로 1위 → `freezer_vision_first_single`이
`2×131=262 ≈ delta 264.8`로 무게 게이트까지 통과 → **zone1도 체리쥬빌레
×2로 오판** (이중 과금 8,000원).

0711 cross-zone penalty가 켜져 있었어도 3중으로 미발동이었다:

1. **오염 창 불겹침** — zone1 W(E)=[594.90, 605.39], zone2 앵커는
   590.95~593.12로 창 하한보다 1.8s 앞. 0711 모델은 "타 존 취출 *장면*이
   찍힌다"를 모델링했지, "취출된 *물건*이 손에 남아 계속 보인다"는
   모델링하지 않았다. 들고 있는 물건의 오염 지속시간은 상한이 없어서
   시간 창으로는 원리적으로 포착 불가.
2. **무게 모호성 게이트(④)가 차단** — delta 264.8을 count_gate ±15 내로
   설명하는 단일 종 해는 체리쥬빌레×2(오차 2.8) **하나뿐**. 진실인
   메로나+쿠즈락(공칭 ~243g)은 오차 ~22g로 게이트 밖. "무게 유일 해 = 신뢰"
   전제가 뒤집힌 케이스 (유일 해가 오염 후보).
3. **재판정도 같은 결론** — class 40을 α=0.5로 강등해도 무게가 맞는
   후보는 그것뿐이라 `freezer_vision_first`가 다시 채택.

→ CLOSE 재판정 계열로는 한계. **트리거 시점에, 비전 증거의 시간 구조로
"이 에피소드에서 꺼낸 물건"과 "이미 들고 있던 물건"을 구분**해야 한다.

## 2. 핵심 관찰 — 시간 구조가 다르다

| | 진짜 취출 후보 | 들고 있던(held) 후보 |
|---|---|---|
| 최초 등장 | 자기 removal 세그먼트 부근 | **프리롤 첫 프레임부터** (자기 change보다 4s 앞 — 이 존의 취출로는 물리적으로 불가능한 시점) |
| 등장 구간 | 취출 동작 전후로 국한 | **영상 전 구간** (손에 계속 있음) |

이슈 #10 실측: class 40은 zone1 영상(디코드 ~470프레임)에서 225표 —
프리롤 포함 전 구간 등장 패턴. 진짜 상품 class 13/44는 66/58표로 자기
세그먼트 부근에만 등장했을 것.

**F6(프레임→벽시계 환산 불신뢰) 비저촉**: 벽시계 환산을 하지 않는다.
"디코드 스트림에서의 상대 위치"만 쓴다 — 프리롤은 스트림 맨 앞
~120프레임(카메라 replay buffer 계약)이라는 *구조적* 사실만 이용.

## 3. 신호 정의

카메라별, 후보 class별로 디코드 프레임 위치(pos, 게이트 스킵 포함 카운트)
기준:

- `head_votes` — 스트림 첫 `HEAD_FRAMES`(기본 30 ≈ 프리롤 첫 1초) 안에서
  받은 표 수. **1프레임 오검출로 오강등되지 않도록 최초 등장 인덱스가
  아니라 표 수(지속 등장)를 센다.**
- `span_ratio` — `(last_pos − first_pos) / decoded_frames`.

**carried-in suspect 판정 (둘 다 필수 — AND)**:

```
suspect(c) := head_votes(c) ≥ HELD_MIN_HEAD_VOTES (기본 5)
           ∧ span_ratio(c) ≥ HELD_SPAN_MIN (기본 0.8)
```

AND인 이유 (§7 S5): 같은 존 연장 병합 에피소드에서 1번째로 꺼낸 상품도
span은 높지만(꺼낸 뒤 계속 손에 있음), 프리롤 초반에는 아직 선반에
있으므로 head 조건이 안 걸린다 — 진짜 취출을 보호하는 쪽이 head 조건.

**강등은 soft**: suspect 후보의 `confidence·vote_count·vote_ratio × β`
(기본 0.4). 목록에서 제거하지 않는다 — 경쟁 후보가 없고 무게가
지지하면 여전히 채택될 수 있다 (§7 S2가 의도된 동작인 이유).

## 4. 구현 지점

| 위치 | 변경 |
|---|---|
| `pipeline._run_vision` | 카메라별 디코드 프레임 카운터(게이트 스킵 포함) → `voting.add_frame(camera, detections, pos=...)` |
| `VotingEnsemble` | (camera, class)별 `first_pos/last_pos/head_votes` 집계. `combine()`이 `VisionCandidate`에 진단 필드 추가: `first_pos_ratio`, `span_ratio`, `head_votes` (frozen 기본값 → 저널/아카이브 하위호환) |
| 강등 적용점 | `combine()` 출력 직후, `_run_vision` 반환 전 — 판정과 이벤트 보존 후보 모두 강등 반영. **원본 수치는 `vote_summary`에 보존** (I8, 사후 진단) |
| `TriggerTrace` | `reason_codes`에 `held_object_demoted:class{N}` 기록 |

트리거 시점 적용인 이유: 신호가 존-로컬이고 자기완결적 — CLOSE까지 기다릴
이유가 없고, 잠정 판정도 함께 좋아진다. 0711 cross-zone penalty(CLOSE,
창 겹침용)와 **상호보완** — 대체가 아니다.

## 6. 단계 배포

1. **A-1 계측 (판정 영향 0)** — 진단 필드만 기록. 실기에서 "진짜 취출
   후보의 head/span 분포"와 "held 후보의 분포"가 실제로 분리되는지 확인
   후 임계 확정. (0711 Phase 1과 같은 태도.)
2. **A-2 강등** — env on. shadow 불필요 (트리거 단위라 diff는 세션
   아카이브의 원본 보존 수치로 확인 가능).

**가드**: 카메라 프리롤이 짧은 트리거(디코드 프레임 < 예상 최소치, 예:
프리롤 공칭 120의 50%)는 위치 의미가 왜곡되므로 해당 트리거에서 강등을
통째로 비활성 (§7 S3).

## 7. 역효과 시나리오 — 적용하면 오히려 정확도가 떨어질 수 있는 경우

| # | 시나리오 | 결과 | 완화 | 잔존 위험 |
|---|---|---|---|---|
| S1 | **느린 취출**: 문 열고 상품을 잡은 채 3.5s+ 머뭇거리다 들어올림 — change보다 한참 전부터 손+상품이 화면에 등장 | 진짜 후보가 suspect로 강등 → freezer는 `loadcell_identity_suppressed` 계열로 NO_DETECTION → **매출 누락** | head 구간을 프리롤 극초반(첫 1초 = change 3~4s 전)으로 좁게 — "그 시점부터 이미 들고 있었다"가 성립하려면 사실상 취출이 이미 끝났어야 함. β soft라 경쟁자 없으면 채택 유지 | **가장 큰 잔존 위험.** A-1 계측에서 실기 빈도 확인 필수. 빈발 시 head 조건에 "표 연속성" 추가 검토 |

요약: 최악 방향은 S1/S2의 "진짜 취출 강등 → 미청구/오판"인데, fail-closed
원칙(과청구가 미청구보다 나쁨, I13/D9)상 **"확신에 찬 이중 과금"을
"보수적 미청구/부분 청구"로 바꾸는 트레이드**라 방향은 옳다. 단 빈도가
관건이므로 A-1 계측 없이 A-2를 켜지 않는다.

## 8. 이슈 #10 재생 기대 결과 (정직한 한계)

class 40(head+span 충족)과 아마 class 27(양 존 영상에 ~130표씩 지속 등장 —
동일 패턴)이 강등되면 `freezer_vision_first`의 단일/조합 탐색이 전부 무게
게이트에서 실패한다 — 진실(멜로나+쿠즈락 공칭 243g)조차 실측 delta
264.8과 ~22g 차이로 게이트(±15) 밖이기 때문. 이후
`vision_first_identity_partial`이 잡아 **PARTIAL 1건(count=1)** 정도가
현실적 기대치다.

즉 이 설계로도 이 세션의 **완전한 진실 복원은 불가** (로드셀 실측이
공칭에서 16~20g 이탈 — §9). 목표는 "8,000원 이중 과금 COMPLETE"를
"보수적 부분 청구/미청구"로 바꾸는 것이고, 그것이 fail-closed 방향이다.

## 9. 부록 — zone1 로드셀 이탈의 원인 (이슈 #10 세그먼트 분석)

세그먼트 합 vs delta:

| zone | 세그먼트 | 합 | delta | 차이 |
|---|---|---|---|---|
| 2 | −130.1, −130.9 | −261.0 | −261.0 | 0 (깨끗) |
| 4 | −233.75, +229.1 | −4.6 | −4.6 | 0 (깨끗) |
| 1 | −58.9, −187.6 | −246.5 | −264.8 | **−18.3** |

zone1만 20g 미만(segment_step 임계 이하)의 plateau 스텝들이 세그먼트로
방출되지 않고 delta에만 반영됐다. 멜로나(공칭 ~75g)가 −58.9로, 쿠즈락
(168g)이 −187.6으로 서로 반대 방향으로 ~16~20g씩 이탈한 것과 합치면 —
**취출 시 손이 선반/이웃 상품을 누르는 접촉 하중(press transient)이
plateau 평균에 섞인 것**과 정합한다. 분석기(`LoadcellAnalyzer`)는 안정
plateau(0.5s간 std ≤ 2g)만 쓰므로 "안정 전 값을 읽는" 구조가 아니고,
진동은 std 게이트 때문에 plateau가 되기 어렵다 — 즉 센서 노이즈가 아니라
**실제로 실린 힘**이 주범일 가능성이 높다. 다른 세션에서 보이는
"+30 세그먼트 후 −실무게" 패턴도 같은 메커니즘 (누름이 0.5s 이상 지속돼
plateau로 성립 → +30 방출 → 눌린 상태에서 취출 → −(무게+30)).

단정하려면 raw 샘플이 필요한데 현재 트리거 페이로드의 loadcell 샘플은
영속화되지 않는다 → **계측 제안**: `|sum(segments) − delta| > tol`인
트리거에 한해 plateau 목록(start/end/mean)과 raw totals를 세션 아카이브
trace에 기록. A-1 계측과 같은 커밋 계열로 처리 가능.
